ESP8266-01S 开门器项目开发规范文档
创建日期：2026-06-29
文档说明：本文档为项目专属开发规范、编码准则、工程管理、编译烧录、版本管控标准，为项目开发强制执行准则，与代码设计文档完全拆分、独立生效、互不混杂。
适用项目：esp8266-opendoor（ESP8266-01S 物联网开门器）
约束范围：所有项目开发、代码修改、功能迭代、固件编译、版本提交行为

---
一、开发环境与硬件规范
1. 目标硬件：ESP8266 ESP-01S 模块（1MB Flash），硬件外设、引脚、内存、存储资源受限，所有开发适配精简硬件特性。
2. 开发框架：Arduino 框架，底层基于 ESP8266 内置简化版 FreeRTOS 任务调度，原生支持多任务并发运行，所有业务基于任务机制开发。
3. 工程配置：通过 platformio.ini 统一管理项目编译、固件、外设配置；默认启用 SPIFFS 简易文件系统；受1MB Flash硬件限制，取消双分区OTA功能，仅支持单分区固件烧录与运行。
4. 任务调度规范：所有业务功能均以 FreeRTOS 独立任务实现，任务之间仅通过队列、事件组完成数据交互与状态同步；禁止跨模块直接调用内部函数，严格保证代码低耦合、高稳定性。
5. 代码配置规范：统一模块接口命名规则，定义全局统一错误码；所有硬件引脚、网络参数、任务栈大小、延时参数、功能阈值等核心配置，全部集中存放于公共配置文件，禁止零散定义、禁止业务代码硬编码。
6. 工程目录规范：严格遵循 PlatformIO 标准目录结构，自研私有驱动、功能模块统一以独立库形式存放于 lib 目录，便于功能复用、后期维护与版本扩展。
7. 开发前置要求：所有开发人员开发前必须通读项目《规则文档.md》《使用指南》，严格遵循项目定制开发要求，未熟读规范禁止开展编码工作。
8. 代码版本注释规范：新建代码文件需在文件头部标注简要功能说明、适配硬件、创建时间；项目初始开发版本为 0.0.1，后续迭代版本每次递增0.1，统一沿用项目现有注释格式，保证格式统一规范。
9. 目录设计原则：所有代码文件分层拆分、职责单一，模块功能独立解耦，适配后期功能迭代、问题精准排查、代码长期维护需求。

二、项目标准文件结构
本结构适配ESP8266-01S硬件资源受限特性，为项目唯一标准源码目录结构，禁止随意新增、删除、修改目录层级与文件命名。
src/
├── main.cpp                # 程序唯一入口，仅负责硬件初始化、系统初始化、创建所有RTOS任务，不编写任何业务逻辑
├── common/                 # 全局通用基础模块（全局底层支撑，所有模块可调用）
│   ├── config.h            # 全局参数配置（引脚、WiFi、MQTT、任务参数、业务阈值等）
│   ├── log.h               # 串口日志打印统一封装（分级日志、统一输出格式）
│   └── err_code.h          # 全局统一错误码枚举定义（标准化异常返回）
├── bsp/                    # 底层硬件抽象层（基础硬件驱动封装，屏蔽SDK差异）
│   ├── bsp_gpio.cpp
│   └── bsp_gpio.h
├── driver/                 # 外设功能驱动层（设备级独立驱动，可插拔）
│   ├── drv_led.cpp
│   └── drv_led.h
├── service/                # 业务服务层（网络、通用系统服务）
│   ├── srv_wifi.cpp
│   ├── srv_wifi.h
│   ├── srv_mqtt.cpp
│   └── srv_mqtt.h
└── app_task/               # FreeRTOS独立业务任务层（所有并发业务任务）
    ├── task_wifi.cpp
    ├── task_mqtt.cpp
    ├── task_uart.cpp
    ├── task_led.cpp
    └── task_headers.h

三、项目编译与烧录规范
1. 构建脚本规范：禁止新建自定义Python构建脚本，项目统一使用根目录 build.py 完成所有编译、烧录、监控、清理操作，统一项目构建流程。
2. 结果校验规范：每次执行构建命令后，必须通过根目录 build.log 日志文件校验编译、上传结果；存在报错、警告、异常信息需立即修复，禁止带问题代码进行硬件测试与功能验证。
3. 通用标准执行命令：
python build.py          # 完整流程：编译代码 + 自动上传固件至开发板
python build.py build    # 仅编译校验代码，不执行烧录，用于代码语法、逻辑预校验
python build.py monitor  # 启动串口监视器，实时查看设备运行日志与业务输出

4. 核心代码约束：main.cpp 仅做系统初始化、硬件初始化、RTOS任务创建，严禁写入任何业务逻辑、功能实现、状态判断代码，保证程序架构标准化、统一化。
5. 开发前置约束：每次编写、修改、迭代代码前，需通读本规范所有条款，严格遵守开发准则，杜绝违规开发。

四、编码行为规范
4.1 编码前逻辑梳理准则
禁止主观假设硬件特性、系统特性与功能逻辑，开发过程中存在疑问必须及时确认；需求存在多种实现方案时，需完整罗列所有可行方案，禁止私自选定方案落地；若需求模糊、逻辑存在歧义，立即停止编码，主动提问澄清，杜绝无效开发、重复开发。
4.2 极简代码开发准则
严格坚持最小功能实现原则，仅开发需求明确要求的核心功能，不新增无关拓展功能、不做过度抽象封装；一次性临时功能无需通用化、可配置化设计；不处理极低概率、无实际业务场景的异常逻辑；持续精简冗余代码，杜绝无效判断、重复逻辑、空函数、未使用变量，避免代码过度复杂、固件臃肿。
4.3 精准修改准则
修改现有代码时，仅改动本次需求对应的代码模块，不随意优化、重构、调整无关代码、注释与格式；严格沿用项目现有代码风格，不自定义编码格式、命名规范；仅删除本次修改产生的未使用导入、变量、函数，原有历史废弃代码仅做注释标注，不擅自批量删除；所有代码修改必须可追溯、对应明确业务需求。
4.4 目标驱动开发准则
开发前明确功能落地标准与验证方式，采用分步开发、分步校验模式，完成一个功能点立即验证一个功能点，杜绝一次性编写大量代码后统一调试；多步骤开发需提前梳理执行计划，明确每一步开发目标、实现逻辑与验证标准，保证功能稳定、逻辑可靠。
4.5 开发工具流程准则
全程基于 PlatformIO 完成项目开发、编译、工程管理，严格遵守项目目录规范、编码规范；所有构建、编译、烧录、监控操作统一通过 build.py 标准脚本执行，禁止使用零散终端命令、自定义工具替代标准流程。

五、Git 版本追溯规范
1. 项目全程使用 Git 进行版本管控，所有提交记录均使用中文清晰、简洁描述变更内容；新增核心功能、重大功能迭代需新建独立开发分支，测试稳定、功能无误后，再合并至主分支，保障主分支代码稳定。
2. 代码修改、功能新增、目录调整、配置变更、bug修复等所有项目操作，均需通过Git提交留存记录，保证版本全程可追溯、可回退、可复盘。
3. 遵循阶段性提交原则，每完成一个可独立运行、可验证的功能模块，立即提交代码，备注清晰说明本次改动目的、新增功能、优化内容与修复点。
4. 无明确指令授权的情况下，禁止删除历史提交记录、强制覆盖代码、执行重置、强制回退等破坏性Git操作，全方位保障项目代码版本安全。
5. 规范配置 .gitignore 文件，自动忽略编译产物、本地临时配置、日志文件、缓存文件、虚拟环境文件等无关内容，杜绝无效文件提交，保证代码仓库整洁。
6. 项目版本规范：项目初始开发版本为 0.0.1，后续所有功能迭代、优化、修复均按照版本规则有序递增，版本迭代可追溯。
