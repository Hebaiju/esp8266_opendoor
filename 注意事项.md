# ESP8266-01S 项目 - 踩坑与注意事项

本文档汇总 ESP8266-01S 开门器项目开发过程中遇到的问题、原因与解决方案，便于后续排查和参考。

---

## 1. PlatformIO 命令未识别

**问题**：
```
pio : The term 'pio' is not recognized as a name of a cmdlet, function, script file, or executable program.
```

**原因**：Windows PowerShell 环境中 PlatformIO 的可执行文件未加入 PATH。

**解决**：
- 使用 `build.py` 脚本调用（脚本内部自动查找 PlatformIO 路径）
- 或使用完整路径：`C:\Users\HEBAIJU\.platformio\penv\Scripts\platformio.exe`

---

## 2. 上传失败：Failed to connect to ESP8266

**问题**：
```
Connecting........_____....._____.....
A fatal error occurred: Failed to connect to ESP8266: Timed out waiting for packet header
```

**原因**：
- ESP-01S 自动下载电路与 esptool 默认协议不兼容
- GPIO0 启动模式识别异常

**解决**：
在 `platformio.ini` 中配置：
```ini
upload_resetmethod = nodemcu
```

---

## 3. 串口监视器看不到正常输出（全是乱码）

**问题**：
- 上传固件成功
- 打开串口监视器后只看到乱码（如 `鈵€l锟斤拷|...`）
- 没有任何正常日志输出

**原因**：
PlatformIO 串口监视器默认模式会通过 **RTS/DTR 引脚** 持续控制 ESP8266 复位，导致设备一直处于复位/重启状态，只输出 boot 阶段的乱码。

**解决**：
使用 `direct` 过滤器模式，禁用串口控制线：
```bash
platformio device monitor --baud 115200 -f direct
```

**build.py 中需要相应修改为 direct 模式**。

---

## 4. 固件启动后无法运行（无任何输出）

**问题**：
- 上传成功
- 用 direct 模式打开串口监视器，设备输出 boot 乱码后无任何后续输出
- LED 不闪烁

**原因**：
**Flash 模式不匹配**。ESP8285（内置 Flash）通常需要 `DOUT` 模式，而 PlatformIO 默认 `QIO` 模式不兼容，导致固件写入后无法正常启动。

**解决**：
在 `platformio.ini` 中配置：
```ini
board_build.flash_mode = dout
```

重新编译上传后，固件可正常运行。

---

## 5. Blinker.begin() 阻塞导致系统挂起

**问题**：
- 调用 `Blinker.begin()` 后整个 setup 流程卡住
- 串口无任何后续输出
- LED、串口命令等任务无法执行

**原因**：
`Blinker.begin()` 内部会阻塞等待 WiFi 连接成功（最长约 20~30 秒），若 WiFi 不可用，行为不确定，可能直接卡死。

**解决**：
采用**延迟连接 + 状态机**：
1. `srv_blinker_init()` 只设置状态为 `DELAY`，记录启动时间
2. `srv_blinker_run()` 循环检测：
   - DELAY 状态：延时 2 秒后切换到 CONNECTING
   - CONNECTING 状态：调用 `Blinker.begin()`，等待 WiFi
   - READY 状态：正常运行 `Blinker.run()`

**要点**：
- Blinker 初始化必须在所有其他系统组件（串口、LED、舵机）初始化完成之后
- 必须先让 WiFi 连接成功，再调用 `Blinker.begin()`

---

## 6. Blinker 库多重定义错误

**问题**：
```
multiple definition of `xxx' (Blinker 库符号)
```

**原因**：
Blinker 库作为单例存在，若在多个 `.cpp` 文件中分别 `#include <Blinker.h>` 并初始化，会导致链接期多重定义。

**解决**：
- Blinker 库的 `#include` 和 `begin()` 初始化**只能在唯一一个文件**中进行（推荐 `srv_blinker.cpp`）
- 其他文件通过 service 层接口调用，**禁止**直接 include `<Blinker.h>`

---

## 7. 自定义错误码与 lwIP 宏冲突

**问题**：
```
redefinition of 'ERR_OK' / 'ERR_TIMEOUT' 等宏
```

**原因**：
ESP8266 SDK 的 lwIP 库已经定义了 `ERR_OK`、`ERR_TIMEOUT` 等同名宏，自定义枚举直接冲突。

**解决**：
所有自定义错误码添加 `APP_` 前缀：
```c
typedef enum {
    APP_OK = 0,
    APP_ERR_TIMEOUT,
    APP_ERR_INVALID,
    APP_ERR_BUSY,
    APP_ERR_FAIL,
} err_code_t;
```
并全局替换所有引用。

---

## 8. ESP8266 Arduino 框架无 FreeRTOS 用户 API

**问题**：
- Arduino 框架基于 Non-OS SDK，无法直接调用 `xTaskCreate`、`xQueueCreate` 等 FreeRTOS API

**解决**：
- 采用 `millis()` + 状态机实现非阻塞多任务调度
- 每个"任务"实现为 `task_xxx_init()` + `task_xxx_run()`，在 `loop()` 中顺序调用
- 严格保持分层：`common → bsp → driver → service → app_task`

---

## 9. ESP-01S 启动引脚限制

**问题**：
- ESP-01S 只有 GPIO0、GPIO1(TX)、GPIO2(LED)、GPIO3(RX) 四个可用引脚
- GPIO0 在启动时必须为高（低电平进入下载模式）

**注意**：
- 舵机控制信号接 GPIO0 时，需确保断电后未连接（不影响下载）
- GPIO2 是板载 LED（低电平点亮），复用时需注意电平逻辑

---

## 10. 串口命令解析要点

**实现注意事项**：
- 必须处理 `\r`、`\n` 两种行结束符
- 必须支持退格键（`\b` 或 `0x7F`）
- 命令缓冲区要做溢出保护（建议 64~128 字节）
- `Serial.available()` 必须在 loop 中轮询，不能用 `delay()` 阻塞

---

## 11. ESP8285 与 ESP01_1M Board 兼容性

**说明**：
- `esp01_1m` board 配置（QIO 模式、默认分区表）原本针对 ESP8266 + 外置 Flash
- 实测设备为 **ESP8285**（内置 1MB Flash）
- 必须修改为 `dout` 模式才能正常启动

---

## 12. 推荐构建命令

```bash
# 仅编译
python build.py build

# 仅上传（自动复位）
python build.py upload

# 串口监视（需 direct 模式）
platformio device monitor --baud 115200 -f direct
```

---

## 13. LED_ACTIVE_LEVEL 与板载 LED

**ESP-01S 板载 LED**：
- 接在 **GPIO2**
- **低电平点亮**（`LED_ACTIVE_LEVEL = 0`）
- 注意上拉电阻，初始化时必须先置高（灭灯状态）再操作

---

## 14. 内存与 Flash 使用控制

当前固件：
- RAM：约 44.6%（36.5KB / 80KB）
- Flash：约 38.0%（388KB / 1MB）

1MB Flash 资源紧张，应避免引入不必要的库。如需添加新功能，优先考虑：
- 复用已有驱动
- 使用 PROGMEM 存储字符串常量
- 减少 `String` 类型使用，改用 `char[]`
